# TopoGlyph 10.0: Documentation Topology

A TopoGlyph extension for designing and growing a `/design` directory of markdown that documents an application's implementation, features, and design ‚Äî and continues to evolve as the codebase and understanding both change.

Builds on 9.0 (Application Design Comprehension). Where 9.0 produces an *ephemeral* mental model from a comprehension walk, 10.0 is the **persistent counterpart**: the comprehension graph crystallized into versioned markdown. The two are designed to compose ‚Äî 9.0 generates the seed, 10.0 specifies how the seed survives, ages, and grows.

## Why a new module is warranted

Persistent documentation has problems that no prior TopoGlyph module addresses:

1. **Decay management** ‚Äî code keeps moving; docs don't. Without explicit drift signals, every doc is *suspected stale* by default after a few months.
2. **Audience layering** ‚Äî implementation docs, feature specs, and design rationales answer different questions for different readers and decay at different rates. They cannot share a schema.
3. **Growth without rewrite** ‚Äî a framework that requires reorganization every time you learn something new will be abandoned. New material must *attach* without restructuring.
4. **Two-way binding** ‚Äî the doc has to know what code it describes; the code has to be discoverable from the doc. Otherwise drift is invisible.
5. **Authority as a first-class field** ‚Äî when two docs disagree, the directory itself must encode who wins, in machine-readable form.

## Modules

### 1. Documentation Substrate Sub-types

Refines `‚ñ§` from 9.0 by **the question each doc kind answers** ‚Äî which determines schema, audience, and decay rate:

| Glyph | Doc kind | Question it answers | Decay |
|---|---|---|---|
| `‚ñ§‚Å±` | Implementation note | *How does this code work?* | Fast (refactor) |
| `‚ñ§·∂Ý` | Feature spec | *What can the user do?* | Medium (UX changes) |
| `‚ñ§·µà` | Design rationale | *Why was it built this way?* | Slow (if "why" is preserved) |
| `‚ñ§·µÉ` | ADR | *What was decided, what was rejected?* | Frozen (append-only) |
| `‚ñ§·µê` | Map / index | *Where do I find X?* | Fast ‚Äî but cheap to regenerate |
| `‚ñ§·µç` | Glossary | *What does this term mean here?* | Slow |

### 2. Typed Documentation Edges

The directory is a graph; edges are typed:

- `‚Üí·µ¢À¢` *is described by* ‚Äî code ‚Üí doc binding
- `‚Üí ≥·µâ·∂Ý` *references* ‚Äî soft "see also"
- `‚Üí ≥·µâ·µÝ` *requires* ‚Äî hard prereq (read-this-first)
- `‚ÜíÀ¢·µò·µñ` *supersedes* ‚Äî newer doc replaces older
- `‚Üí·µÉ·µê·µñ` *amplifies* ‚Äî expands on (high-level ‚Üí detail)
- `‚Üí·∂ú·µí‚Åø·∂Ý` *conflicts with* ‚Äî open contradiction needing resolution
- `‚Üí·µâÀ£` *exemplifies* ‚Äî points to a concrete example doc

The `‚ÜíÀ¢·µò·µñ` and `‚Üí·∂ú·µí‚Åø·∂Ý` edges encode authority *in the graph itself*, not in prose buried at the top of an architecture file.

### 3. Lifecycle States

Every document has exactly one state at any time:

- `‚óØ` Draft (incomplete; do not trust)
- `‚óè` Current (authoritative)
- `‚óê` Fading (probably outdated, unverified)
- `‚óë` Confirmed-stale (verified outdated, kept for history)
- `‚óç` Superseded (see `‚ÜíÀ¢·µò·µñ` target)
- `‚äï` Born-from-question (was `‚ñ§·µÝ`, now answered ‚Üí `‚ñ§·µà` or `‚ñ§‚Å±`)
- `‚äñ` Folded (content moved elsewhere; this is a redirect stub)

### 4. Growth Operators

Named *moves* that mutate the directory. Each is a primitive that should appear in a CHANGELOG so the directory's evolution is itself navigable:

- `‚äû` **Spawn** ‚Äî section grew enough to extract into its own file
- `‚äÝ` **Merge** ‚Äî two related notes folded into one
- `‚äü` **Split** ‚Äî one overgrown doc becomes multiple
- `‚•Å` **Refactor** ‚Äî restructure without semantic change
- `‚Üó` **Promote** ‚Äî informal scratch note ‚Üí formal `‚ñ§·µà` or `‚ñ§·µÉ`
- `‚Üò` **Archive** ‚Äî `‚óè` ‚Üí `‚óë`/`‚óç` and move to `archive/`
- `‚äï` **Resolve** ‚Äî `‚ñ§·µÝ` answered, becomes `‚ñ§·µà`/`‚ñ§‚Å±`/`‚ñ§·µÉ`
- `‚èö` **Anchor** ‚Äî extract a load-bearing fact into `INVARIANTS.md`

### 5. Freshness / Drift

The mechanism that keeps the directory honest:

- `‚åÅ` *last-touched* (timestamp; trivially derivable from git)
- `‚åá` *last-verified* (someone confirmed it still matches reality ‚Äî **not** the same as last-touched)
- `‚èö` *anchor* (a declared invariant; should not drift, ever)
- `‚èõ` *drift signal* (the bound code changed after `‚åá` ‚Äî auto-generated by lint)
- `‚äô` *test-backed* (the doc's central claim has an automated test)

`‚åá` is the key innovation. Most doc decay happens because *editing* a doc is easy but *re-reading and confirming* a doc is invisible work. Making `last_verified` a separate field promotes the verify action to a first-class event.

### 6. Code‚ÜîDoc Binding

- `‚Ü¶` *binds* ‚Äî `‚ñ¶path/to/code ‚Ü¶ ‚ñ§path/to/doc` declares two-way responsibility
- `‚Ü§` *back-binds* ‚Äî code points back at the doc (via comment or path convention)

A binding plus a `‚åá` plus `git log -- <bound paths>` is a complete drift detector.

## Front-Matter Schema (machine-readable)

Every file in `/design` carries YAML front-matter so the directory can be linted, queried, and drift-checked:

```yaml
---
id: I-pipeline                  # stable handle, used in links
title: Test Run Pipeline
type: implementation            # implementation | feature | design | adr | map | glossary | runbook | question | constraint
state: current                  # draft | current | fading | stale | superseded
last_touched: 2026-05-08        # auto from git
last_verified: 2026-05-01       # MANUALLY bumped when read-and-confirmed
binds:                          # ‚Ü¶ ‚Äî the code this doc describes
  - apps/server/src/services/pipeline/**
  - apps/server/src/services/agents/**
references: [I-mcp, ADR-0017]   # ‚Üí ≥·µâ·∂Ý
requires: [GLOSSARY]            # ‚Üí ≥·µâ·µÝ
supersedes: []                  # ‚ÜíÀ¢·µò·µñ

```

The schema is small on purpose. Two fields earn their keep on day one: `binds` (so drift is detectable) and `last_verified` (so verification is a first-class action).

## The Process

### Phase 1 ‚Äî Seed (one sitting, ~1 hour)

Run the 9.0 comprehension walk. Persist its output as the initial directory:

| Walk output | Becomes |
|---|---|
| Subsystem map (Step 2)              | `implementation/INDEX.md` listing one `I-*` stub per subsystem |
| Typed boundaries (Step 3)           | A section in `README.md` + entries in `GLOSSARY.md` |
| Spine trace (Step 4)                | `I-pipeline.md` (or whatever the spine is named) |
| Reconciled invariants (Step 7)      | `INVARIANTS.md` with each `‚ü¶‚óà‚üß` as an anchor |
| `?`-list (Step 8)                   | `OPEN-QUESTIONS.md` |
| Authority precedence (Step 1)       | A "Precedence rules" section in `README.md` |

A minimum viable `/design` is ~6 files, none of them deep. The seed is intentionally undersized ‚Äî the framework grows by `‚äû spawn`, not by writing thick files up front.

### Phase 2 ‚Äî Grow

Three triggering events, each tied to an operator:

1. **A new question keeps coming up** ‚Üí add to `OPEN-QUESTIONS.md`. When answered, `‚äï resolve` it into a new `‚ñ§·µà`/`‚ñ§‚Å±`/`‚ñ§·µÉ`.
2. **A section in an existing doc gets long enough to have its own readers** ‚Üí `‚äû spawn` it.
3. **A decision is made that closes off alternatives** ‚Üí write an `‚ñ§·µÉ` ADR. **Append-only** ‚Äî you don't edit ADRs, you supersede them.

Don't write docs you don't need yet. Empty subdirectories with an `INDEX.md` stub are fine.

### Phase 3 ‚Äî Maintain (continuous, ambient)

Three small disciplines, each tied to a glyph:

- **`‚åá` Verify on read.** When you read a doc to do real work and find it accurate, bump `last_verified` in the same commit as the work. Verification piggybacks on actual reading; it's not a separate chore.
- **`‚èõ` Drift watch.** A small lint script: for each doc, compare `git log --since=<last_verified> -- <binds>`. Any output ‚Üí mark `state: fading`. Can run weekly in CI.
- **`‚èö` Promote invariants up.** When a fact is load-bearing across multiple docs, lift it into `INVARIANTS.md` and replace the in-place mentions with anchor references. Single-source-of-truth, by accumulation rather than upfront design.

## Why this framework can evolve

Three structural properties:

1. **Append-only modes** for the parts that decay if mutated. ADRs and `INVARIANTS.md` are append-only / supersede-only. You never lose history; you layer new layers on top ‚Äî exactly like idempotent additive boot migrations at the schema layer.

2. **Named structural operators** mean restructurings have vocabulary. "I'm going to `‚äü split` `I-agents` into four because the file got too long" is a one-line CHANGELOG entry, not a discussion. The operators are the API of evolution.

3. **The seed is undersized on purpose.** Most documentation projects fail because the upfront structure is overbuilt; sections sit empty and atrophy. Here the seed is ~6 small files. Everything else is grown by `‚äû spawn` from things people actually wrote in the course of doing real work. **The directory cannot exist ahead of the comprehension; it can only crystallize behind it.**
